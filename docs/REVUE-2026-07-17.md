# Revue complète du projet — 2026-07-17

Revue de l'état réel du code (backend `src/`, frontend `web/src/`, infra) par
rapport aux règles de `CLAUDE.md`, `docs/UI_UX.md` et aux affirmations de
`TODO.md`. Méthode : lecture intégrale du backend, review frontend déléguée et
vérifiée, exécution de `bun run check` (typecheck + lint + tests + build :
**tout passe**).

## Synthèse

Le socle est sain : collecte de statut découplée des clients, arrêt propre
complet, ping borné par watchdog, identité device protégée (0600), secrets et
données correctement hors de Git, agrégation SQL en buckets. Les problèmes
restants sont concentrés sur trois axes : **validation des entrées et de la
configuration** (P0 déjà identifié, toujours ouvert), **robustesse du client
gateway** (handshake sans watchdog), et **fiabilité du chat** (protocole
fragile, connu). Aucune trahison des règles d'architecture n'a été constatée.

## Constats backend

### Critiques

1. **`env.ts:18` — le garde `BIND_HOST` est contournable.** Seul `"0.0.0.0"`
   est refusé ; `"::"` (wildcard IPv6, équivalent exact) ou n'importe quelle IP
   LAN passent. Scénario : `BIND_HOST=::` expose le dashboard sur toutes les
   interfaces, exactement ce que la règle veut empêcher. → Allowlist
   loopback/plages Tailscale (déjà prévu en P0, à faire en premier).

2. **`src/gateway/client.ts:90-115` — handshake sans watchdog.** Si le socket
   TCP s'ouvre mais que la gateway n'envoie jamais `connect.challenge` (gateway
   gelée, proxy muet), le client reste indéfiniment ni connecté ni en
   reconnexion : `onclose` ne se déclenche pas, aucun timer ne surveille la
   phase de handshake. Le chat et les statuts OpenClaw restent « déconnectés »
   sans récupération possible sans redémarrage. → Timer de handshake qui ferme
   le socket après N secondes sans `connect-ok`.

3. **`index.ts:174` — `?hours=abc` produit `NaN` en cascade.** `Number("abc")`
   → `NaN` traverse `Math.min/max`, donne `since=NaN` et `bucketMs=NaN`
   transmis tels quels à SQLite (`db.ts:59`). Résultat non défini (erreur ou
   réponse vide) au lieu d'un 400 propre. Confirmé ouvert (P0 « valider toutes
   les entrées »).

4. **`index.ts:248` — `getHistory().then(...)` sans `.catch`.** Si la promesse
   rejette (déconnexion gateway entre-temps) : unhandled rejection, et un
   `ws.send` possible sur socket déjà fermé. Le cas `messages: null` est en
   revanche bien absorbé côté front (`parseHistory`). Ajouter `.catch` et un
   garde `readyState`.

### Moyens

5. **`log-tailer.ts:71-84` — spam d'erreurs quand la gateway est déconnectée.**
   Le poll continue toutes les 2 s et chaque `getLogs` rejette immédiatement
   (« gateway not connected ») → un événement `error` toutes les 2 s vers tous
   les clients SSE logs. → Suspendre le tail tant que `gateway.isConnected`
   est faux, reprendre sur l'événement `status`.

6. **`env.ts:27` — `PORT` non validé** (`PORT=abc` → `NaN` passé à
   `Bun.serve`), **`AUTH_TOKEN=change-me` accepté** (la valeur d'exemple de
   `.env.example` fonctionne telle quelle). P0 déjà prévu, confirmé ouvert.

7. **`index.ts:48,242` — comparaison de tokens non constant-time** (`!==` et
   `===`). Faible risque réel derrière Tailscale, mais la comparaison sûre est
   déjà exigée par la spec P3 de `/notify` : la centraliser maintenant
   (`crypto.timingSafeEqual`) sert les deux.

8. **`index.ts:257` — pas de taille maximale sur le texte du chat.** Un client
   authentifié peut relayer un message arbitrairement gros vers la gateway.
   Borne simple (par ex. 8 Ko) + réponse d'erreur explicite.

9. **`network.ts:89-93` — passerelle par défaut mise en cache pour toujours.**
   Si la route par défaut change (bascule réseau), les pings « orange »
   continuent vers l'ancienne IP jusqu'au redémarrage. → TTL ou invalidation
   sur échec répété.

10. **`gateway/client.ts:117-121` — backoff sans jitter, protocole non
    vérifié, pas de suivi `seq`.** Conforme au constat déjà écrit en P0 ;
    confirmé à la lecture, rien n'est fait.

11. **`openclaw-status.ts:178-180` — `whatsappHealthy` reste faux tant que
    `lastError` est non vide**, même si le canal est reconnecté et fonctionnel
    depuis. Pondérer avec la fraîcheur de l'erreur ou l'activité récente.

### Mineurs

- `status-collector.ts:72` : `refresh()` pendant un cycle en cours est ignoré
  (pas de re-run mis en file) — un changement d'état gateway pendant une
  collecte attend le cycle suivant.
- `db.ts:68` : `MIN(ok)` marque tout un bucket en échec pour une seule perte —
  pessimiste et surtout aucune donnée de taux de perte (P1 ouvert).
- `launchd/com.clawdeck.server.plist` : le dossier de logs n'est pas créé
  automatiquement et il n'y a pas de `ThrottleInterval` (P4 ouvert).
- Couverture de tests réelle : 16 tests unitaires ciblés (network, collector,
  log-tailer, logs, openclaw-status, checks). Rien sur `env`, `db`,
  les routes HTTP/WS ni `gateway/client` — les zones où sont les bugs ci-dessus.

### Points solides (à préserver)

- Boucle de collecte unique et découplée, snapshot poussé aux SSE sans sonde
  par client, client lent ne gardant que le dernier snapshot (`index.ts:56-97`).
- Arrêt propre exhaustif (`index.ts:283-329`).
- Ping borné trois fois (`-t`, `-W`, watchdog Bun) avec drain des flux.
- Identité Ed25519 persistée en 0600, dossier 0700, empreinte vérifiée au
  rechargement.
- `.env` et `data/` réellement absents de Git (vérifié via `git ls-files`).
- Logs bornés partout : 500 entrées par batch SSE, 4 000 caractères par
  message, `maxBytes` côté RPC.

## Constats frontend

Review déléguée puis contre-vérifiée : les quatre constats critiques ont été
confirmés ligne à ligne dans le code.

### Critiques

1. **`useChat.ts:145` — la déduplication du miroir supprime des messages
   légitimes.** Un `session-message` est écarté si un message de même rôle et
   même texte existe **n'importe où** dans la conversation. Envoyer deux fois
   « ok » depuis WhatsApp dans la journée → le second n'apparaît jamais ;
   idem pour deux réponses assistant identiques. Cette comparaison globale ne
   sert qu'à absorber l'écho de l'envoi optimiste. → Dédupliquer l'écho
   uniquement contre les messages `local-*` récents non réconciliés et
   s'appuyer sur `stableId`/`runId` (lignes 143-144, déjà corrects) pour le
   reste. (Recoupe le P2 existant du TODO — confirmé réel.)

2. **`useChat.ts:204-209` — fermeture `1008 unauthorized` traitée comme une
   coupure réseau.** `onclose` ignore `ev.code` ; le backend ferme avec 1008
   (« unauthorized » ou « auth timeout », `src/index.ts:230,252`) et le hook
   re-tente toutes les 3 s avec le même token rejeté, indéfiniment. Le SSE
   status nettoie le token dans la plupart des cas, mais si seul le WS échoue,
   la boucle tourne sans fin — l'anti-pattern interdit par UI_UX.md §6.
   → Inspecter `ev.code === 1008`, exposer un état `unauthorized`, ne pas
   re-tenter.

3. **`vite.config.ts:12-17` — le chat est cassé en dev.** Le proxy `/api` ne
   relaie pas les WebSockets : `ws://localhost:5173/api/chat/ws` n'atteint
   jamais le backend. Correction d'une ligne : `ws: true`. (Prod non
   affectée : le front y est servi par Hono.)

4. **`usePingHistory.ts:13-30` — race sur le changement de période.** Pas
   d'`AbortController` ni de garde d'obsolescence : en passant de 24 h à 7 j,
   la requête 24 h encore en vol peut se résoudre après celle de 7 j et
   écraser l'état — graphe étiqueté « 7 j » affichant les données 24 h
   pendant 30 s. → Abort dans le cleanup de l'effet.

### Moyens

5. **`useChat.ts:221-230` — message optimiste jamais réconcilié ni marqué en
   échec** : aucun accusé d'envoi ; un message peut rester affiché comme parti
   alors que la gateway l'a refusé. Viole UI_UX.md §5 (chat et actions).
   Nécessite le `clientMessageId` + réponse succès/erreur déjà prévu en P2.
6. **`App.tsx:131-133,229-231` — données périmées affichées vertes** : quand
   le SSE se fige, les cartes gardent leur tone « Opérationnel » ; l'heure de
   mise à jour n'est même pas recalculée (pas de timer). UI_UX.md §4 exige
   l'inverse. → Badge d'âge recalculé chaque seconde + tones dégradés au-delà
   d'un seuil.
7. **Reconnexions sans backoff** (`useChat.ts:208`, `useStatusStream.ts:70`,
   `useLogStream.ts:88`) : retry fixe 3 s pour toujours (~1 200 req/h par flux
   et par onglet, backend éteint). → Backoff exponentiel plafonné + relance
   sur `online`/`visibilitychange`.
8. **`LatencyChart.tsx:72-87` — ligne tracée à travers les trous de
   collecte** : le bucketing SQL omet les buckets vides (clawdeck éteint,
   veille) et le path relie artificiellement les deux points qui encadrent le
   trou — interdit par UI_UX.md §5. → Couper le path si l'écart entre points
   dépasse ~1,5 × `bucketMs`.
9. **`useChat.ts` — historique chat non borné en mémoire** (contrairement aux
   logs, plafonnés à 500) : `messages` et `runIdToMessageId` grossissent sans
   limite avec le miroir WhatsApp. → Cap glissant.
10. **`App.tsx:343-368` — changer d'onglet détruit l'état du chat** :
    démontage de `ChatPanel` → WS fermé, streaming/tool calls/optimistes
    perdus ; au retour, l'historique ne restaure que le texte. → Remonter
    `useChat` au niveau App ou garder les panneaux montés/masqués.
11. **401 mal géré sur logs et history** (`useLogStream.ts:43`,
    `usePingHistory.ts:19`) : affiché brut « HTTP 401 » et re-tenté, ou ignoré
    en silence, au lieu de s'arrêter proprement comme le fait
    `useStatusStream.ts:34-39`.
12. **`LatencyChart.tsx:139-145` — tooltip au survol uniquement**, valeurs
    inaccessibles au clavier, pas de synthèse textuelle de période (moyenne /
    max) pourtant demandée par UI_UX.md §5.

### Mineurs

- `App.tsx:183` : variante Tailwind `xs:` inexistante (classe morte) ;
  `App.tsx:63` : `refresh` inutilisé ; `App.tsx:156,199` : deux `nav` avec le
  même `aria-label` (landmarks dupliqués).
- `useChat.ts:25,76,100,131` : `any` aux frontières WS, aucune validation des
  frames entrantes (P2 du TODO, confirmé).
- `ChatPanel.tsx:104` : `scrollTo({ behavior: "smooth" })` non neutralisé par
  `prefers-reduced-motion` (la règle CSS ne couvre pas un behavior explicite).
- `useStatusStream.ts:47-64` : flux terminé proprement → « Temps réel actif »
  affiché à tort pendant 3 s.
- `LogsPanel.tsx:72` : point de statut sans `aria-hidden` ;
  `LogsPanel.tsx:119` : « tail tronqué » collant après un `reset`.
- `index.css:5-6` : `Inter` en tête de pile contre la prescription système
  d'UI_UX.md §3 (non bundlée, effet limité).
- `web/package.json` : `tailwindcss`/`@tailwindcss/vite` en `dependencies` au
  lieu de `devDependencies` ; `react-markdown` sans `remark-gfm` (tableaux et
  listes de tâches de l'agent rendus en texte brut).

### Points solides (front)

- Auth SSE par header via fetch + ReadableStream (jamais de token en URL),
  gestion 401 du flux status exemplaire (arrêt du retry, garde anti-boucle).
- Pas de XSS : `react-markdown` sans `rehype-raw`, payloads d'outils rendus
  échappés et bornés (8 000 caractères, repliés).
- Parsing SSE robuste (frames coupées bufferisées), logs plafonnés à 500 avec
  dédup, pause, filtre et recherche.
- États UI largement couverts : vides expliqués, `aria-*`, `focus-visible`,
  `lang="fr"`, responsive 320 px, reduced-motion CSS, autoscroll suspendu
  quand l'utilisateur relit.

## Fonctionnalités à ajouter (recommandations)

Par ordre de valeur pour un dashboard d'exploitation :

1. **Badge d'âge / péremption sur toutes les cartes** — « il y a 12 s »
   recalculé en continu, tones dégradés au-delà d'un seuil. Corrige le constat
   front 6 et matérialise la règle « ne jamais garder un vert sans âge ».
2. **Statistiques de période sous le graphe réseau** — p50/p95 et % de sondes
   en échec par série (les buckets `ok=0` existent déjà) : une coupure
   nocturne est invisible sur une courbe de latence moyenne.
3. **Bandeau d'anomalies récentes** sous le résumé global — dernier échec de
   sonde ou déconnexion avec horodatage, même après retour au vert. S'inscrit
   dans la hiérarchie « état global → anomalies → détails ».
4. **Reprise des logs par curseur** (`Last-Event-ID` côté SSE) — aujourd'hui
   tout ce qui est émis entre deux connexions est perdu.
5. **Actions de pilotage bornées avec confirmation** (relancer le canal
   WhatsApp, forcer un re-check, `chat.abort`) — le CLAUDE.md annonce un
   dashboard qui « pilote » OpenClaw, le front n'expose aucune action ; à
   faire en respectant le pattern pending/accusé/échec et la règle « pas de
   commande système générique ».
6. **Phase 3 (notifications ntfy)** reste la prochaine grande fonctionnalité
   déjà spécifiée dans le TODO — rien à re-spécifier, la spec P3 est bonne.
